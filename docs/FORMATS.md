# 파일 포맷 분석 (Nobunaga's Ambition: Shinsei, Switch)

이 문서는 한글 패치를 만들며 리버스 엔지니어링한 게임 파일 포맷을 정리한 것입니다.
게임 원본 데이터는 포함하지 않습니다. 아래 도구들은 모두 `tools/`에 있습니다.

- 타이틀 ID: `01007AB012872000` (일반 게임, 일본판) + `01007AB012872001` (파워업키트, §7 참고)
- 대상 버전: **1.1.5**(파워업키트 적용) — 베이스 + 업데이트 BKTR 병합본

---

## 0. 게임 실행 데이터 = 베이스 + 1.1.4 업데이트 병합

게임은 베이스 NCA에 1.1.4 업데이트 NCA를 BKTR 델타 패치로 얹어 실행합니다.
따라서 베이스만 추출하면 안 되고 **병합 RomFS**를 봐야 합니다.
`tools/extractor/`(LibHac 0.17.0, .NET 6)가 prod.keys + title.keys로 두 NCA를
복호화·병합해 추출합니다. (키는 저장소에 없음 — 본인 콘솔 덤프 사용)

병합 romfs 파일 수 208개. 언어별 로컬라이즈 대상은 `MSG/{JP,SC,TC}/*`와
`RES_{JP,SC,TC}/res_lang.bin` 둘뿐입니다.

## 1. KOEI LINKDATA 아카이브

```
'LINK' (4)
u32 count
u32 toc_offset        ← 목차 시작 오프셋 (외부 컨테이너=0x10, 내부=0x20 인 경우 있음)
u32 ...
[toc_offset] : (u32 entry_offset, u32 entry_size) × count   ← 엔트리 기준 상대
```

- **주의**: `toc_offset`을 헤더의 offset 0x08에서 읽어야 함(하드코딩 금지).
  외부 res_lang 컨테이너는 TOC가 0x10, 내부 엔트리들은 0x20에 있어 파서가 흔히 어긋남.
- **⚠️ 내부(중첩) LINKDATA의 데이터 서브엔트리 오프셋은 블롭 기준 16바이트 정렬 필수.**
  정렬이 깨지면 게임 로더가 텍스처 로드에 실패하고 4×4 평균색 플레이스홀더(검은 화면)로 폴백함.
- 파서: `tools/linkdata.py`, `tools/koloc.py`(link_children)

## 2. KT 압축 래퍼

```
01 01 (2) + 해시/플래그 (6) + u64 dec_size + u64 comp_size + LZ4 단일 블록
```
- 헤더 24바이트, 이후 raw LZ4 블록. `lz4.block.decompress(data, uncompressed_size=dec_size)`
- 재압축 시 헤더 0~7바이트(매직+해시) 보존, dec/comp 크기 갱신.
- 파서: `tools/kt_unwrap.py`

## 3. NSO0 실행파일 (exefs/main)

- 세그먼트 3개(.text/.rodata/.data)가 각각 LZ4 압축(flags 0x3f), 세그먼트별 SHA256.
- 패치 절차: 압축 해제 → .rodata 문자열 교체(원본 UTF-16 바이트 길이 이내, 널 패딩)
  → 세그먼트별 SHA256 재계산 → LZ4 재압축 → NSO0 헤더 재작성.
- 도구: `tools/nso_decompress.py`, `tools/patch_main.py`

## 4. MSG 텍스트 (`MSG/{lang}/`)

3파일 모두 KT 래퍼 → 오프셋 테이블 컨테이너.

### strdata.bin / ev_strdata.bin — 오프셋 테이블 문자열
```
컨테이너: u32 count + (u32 off, u32 size) × count
각 섹션: 헤더(0x14B) + u32 offset[n](테이블 기준 상대) + null 종료 UTF-16LE 풀
```
- 섹션 헤더 = `u32 0x134C58` + `u16 크기하위16, u16 1` + `u16 문자열수 n, u16 4` + `u32 0x14` + `u32 0xFFFFFF00`
- 컬러 태그: `<ESC>C?` (U+001B + 'C' + 한 글자, Z=리셋)
- strdata 섹션: 0=성씨/이름/관직, 1=UI, 2=무장열전, 3=명품해설, 4=크레딧

### msggame.bin — 바이트코드
- 18섹션, `u32 n + u32 offset[n]` + 바이트코드 엔트리
- 텍스트는 `07 07 01` ~ `07 07 02` 사이 UTF-16LE 런(홀수 오프셋 가능, 정렬 없음)
- 기타 옵코드: `01 4A`+u32=ID참조, `02`+ASCII/`04 3D`+숫자=조건, `06 7C`='|', `1B 43 xx`=컬러, `05 05 05`=종결
- JP/SC/TC 섹션·문자열 개수 1:1 → 슬롯 단위 교체 가능
- 도구: `tools/msg_parse.py`, `tools/msg_export.py`, `tools/apply_translations.py`, `tools/msg_verify.py`

## 5. G1N 비트맵 폰트 (`RES_JP/res_lang.bin` 엔트리 6·7)

매직 `_N1G0000`. 엔트리 6 = 메인 48px, 엔트리 7 = 보조 24px.

```
헤더: +0x08 총크기, +0x0C 첫섹션오프셋, +0x10 팔레트당 엔트리수(16),
      +0x14 비트맵풀 시작오프셋, +0x18 팔레트 개수, +0x1C 섹션 수,
      +0x20 섹션 오프셋 u32×N, 이후 RGBA 팔레트(64B)×팔레트개수
섹션(=스트라이크): charmap u16[0x10000] (문자→글리프ID, 0=없음)
      + 글리프 레코드 12B × (maxID+1)
글리프 레코드 12B: 8바이트 메트릭 + u32 비트맵오프셋(풀 기준 상대)
```

### 메트릭 8바이트
`[b0=논리폭/전진폭, b1=48, b2=0, b3=48, b4=전진폭(=b0), b5=-(저장폭/2) 부호화, b6=0, b7=48]`
- **b5 = 256 - (저장셀폭/2)**. 232→48폭 저장, 244→24폭 저장.
- 섹션2(가변폭 라틴/가나 378자)가 b0(논리폭)만 15~47로 다르게 동작함을 실증.

### 비트맵
- 4bpp 팔레트 인덱스, **행 피치 = ceil(b0/2) 바이트** (⚠️ 폭 바꾸면 반드시 행 재패킹)
- **니블 순서: 짝수 픽셀 = 상위 니블**. 반대로 쓰면 인게임에서 획에 1px 틈+가로 점선.
- 48×48 셀 = 0x480 바이트.
- charmap이 BMP(u16)라 한글 음절(U+AC00–D7A3) 표현 가능.

### 자간 조정 (중요)
게임 텍스트 렌더 경로가 2개:
1. **가변폭**(대사창 등): 글리프 쿼드=b0폭 1:1, 펜 전진=b4
2. **고정칸**(일부 창): b0폭 텍스처를 고정 48px 칸에 스트레치

→ 자간을 안전하게 줄이려면 **b4(전진폭)만** 변경. b0/비트맵을 줄이면
①행 피치 재패킹 필요 + ②고정칸에서 글자가 가로로 늘어남(이중 함정).
- 도구: `tools/g1n_analyze.py`, `tools/g1n_render.py`, `tools/g1n_extend.py`(한글 확장), `tools/font_tighten3.py`(b4만 조정)

## 6. G1T 텍스처 (Switch)

매직 `GT1G` ver0600. res_lang·res_grp 등 아카이브 내부에 다수.

```
헤더: +0x0C 텍스처테이블오프셋, +0x10 텍스처수, 테이블=u32 offset×n
텍스처 헤더: +1 = 포맷, +2 = packed 치수, +8 u32 확장헤더크기 후 데이터
```
- 포맷: `0x59`=BC1, `0x5B`=BC3, `0x5C`=BC4, `0x5F`=BC7, `0x01`=BGRA8
- packed 치수: **하위 니블 = 폭 log2, 상위 니블 = 높이 log2**
- **스위즐 없음(리니어)** — Tegra 디스위즐 불필요.
- 디코드: BC1/3은 DDS 래핑→Pillow, BC4/7은 texture2ddecoder. 인코드(BC3): 자작 `tools/bc_encode.py`(Pillow는 BC 인코딩 불가).
- **로컬라이즈된 UI가 들어있는 res_lang 숨은 엔트리**(내부 LINK TOC를 0x20에서 읽어야 나옴):
  - e5 = 시스템/내비 버튼 아틀라스, e8 = 원형 명령 버튼, e12 = 전투 군평정,
    e13 = 합전 배너, e16 = 튜토리얼 다이어그램, e24 = 타이틀
- 도구: `tools/g1t_decode.py`, `tools/catalog_all_g1t.py`, `tools/extract_hidden.py`,
  `tools/koloc.py`(디코드+한글 fit 렌더+BC3 재인코드+재조립), `tools/loc_atlas.py`, `tools/e5_loc.py`

### 자동 버튼 검출이 놓치는 비표준 모양 라벨
`e5_btns.json`(e5의 버튼 좌표 목록)은 컨투어/블롭 검출로 자동 생성됐는데, **표준
둥근사각형 버튼과 모양이 많이 다른 요소**(예: 뾰족한 육각형 `開戦`(개전) 버튼,
회전된 소형 상태 배지 `中止/失敗/達成`(e8))는 검출 대상에서 빠질 수 있음 — 발견하려면
전체 텍스처를 직접 렌더링해 육안 스캔하거나, 좁은 좌표 범위에 한정해 컨투어 검출을
재실행해야 함. 회전된 배지는 축 정렬 밴드로 잉크 영역을 특정하면 모서리가 잘려
잔상이 남으므로, 알파 채널을 안쪽으로 침식(erode)해 배지 내부 전체를 지우는 방식이
안전함. 도구: `tools/fix_missed_labels.py`, `tools/repack_missed_labels.py`

## 7. 파워업키트(872001) 이중 프로그램 구조

파워업키트가 설치된 NSP를 열어 보면 프로그램이 **두 개** 들어있습니다.

| | 타이틀 ID | 관계 |
|---|---|---|
| 일반 게임 | `01007AB012872000` | base + 1.1.5 업데이트(BKTR) |
| 파워업키트 | `01007AB012872001` | **패치만 존재, 자체 base 없음** — 872000 base 위에 얹는 BKTR 델타 |

### 폴더가 통째로 분리됨
파워업키트 프로그램의 romfs는 일반 게임과 **다른 이름의 폴더**를 씁니다(`_PK` 접미사):

```
01007ab012872000/romfs/   MSG/{JP,SC,TC}/*.bin        RES_JP/res_lang.bin       (§4, §5)
01007ab012872001/romfs/   MSG_PK/{JP,SC,TC}/*.bin      RES_JP_PK/res_lang_pk.bin  RES_JP_PK/res_lang_exp_pk.bin
                           (+ FLOW_PK/ PARAM_PK/ RES_PK/ SCENARIO_PK/ SNDRES_PK/ ...)
```

즉 실기에서 이미지(res_lang, 두 프로그램이 **바이트까지 완전히 동일**)는 872000 쪽
모드가 적용되어 한글로 나오는데, 텍스트(MSG_PK, 872000의 MSG와 다른 내용/다른 파일명)는
872000 모드가 안 닿아 일본어로 남는 비대칭이 생김. **텍스트를 고치려면 872001 쪽에
따로 패치를 넣어야 함.**

### ⚠️ PC 추출로는 안 나옴 — 실기 덤프 필수
872001은 자체 base NCA가 없어서(패치만 존재) hactool/LibHac 0.17 계열로 병합 추출을
시도하면 잘못된 베이스(구버전 872000)에 얹혀서 `MSG_PK`/`RES_JP_PK`가 아예 안 보임.
**DBI/nxdumptool 등으로 실기에서 직접 덤프**해야 정확한 구조가 나옴(Program 0 = 872000,
Program 1 = 872001로 표시됨).

### MSG_PK 포맷
`msgdata.bin msgev.bin msggame.bin msgui.bin msgbre.bin msgire.bin msgstf.bin`(스태프=
크레딧, 원문 유지) 7파일. 오프셋 테이블 문자열/`msggame.bin`은 §4의 strdata/msggame과
**동일 포맷** — 파서 재사용 가능. 내용은 일반 게임 MSG와 상당수 겹치므로(공유 대사·
UI 문자열), 기존 번역 사전을 원문 문자열로 매칭하면 대부분 무료로 커버됨
(`tools/build_msgpk.py`). PUK 전용 신규 문자열만 추가 번역이 필요.
- 더미(미사용) 데이터: `〜ダミー説明`/`〜ダミールビ` 패턴의 문자열은 화면에 표시되지
  않는 미사용 슬롯 placeholder — 번역 대상에서 제외해도 무방.
- 시나리오 검색용 순수 가나 키: `カワナカジマノタタカイマエ` 같은 순수 가타카나
  문자열은 시나리오 이름의 읽기(요미가나) 검색/정렬용 내부 키로 추정 — 번역하면
  내부 매칭이 깨질 수 있어 원문 유지가 안전.
- 도구: `tools/find_puk_new_strings.py`(신규 문자열 탐지+더미/크레딧 분류),
  `tools/prep_puk_batches.py`(번역 배치화), `tools/build_msgpk.py`(최종 재조립),
  `tools/msgpk_normalize.py`(전각→반각 정규화)

### res_lang_pk.bin 폰트 — 크기 불변 제자리 삽입이 필수
`res_lang_pk.bin`(LINK 27엔트리)의 엔트리 16·17이 G1N 폰트(§5와 동일 구조, 일반 게임
res_lang의 엔트리 6·7과 글리프 수·섹션까지 완전 동일). 여기서 §5처럼 폰트를 통째로
더 큰 한글 폰트로 **교체**하면(압축 해제 크기가 원본보다 커짐) **872001이 부팅 전에
죽는다** — 크래시 리포트도 안 남을 정도로 이른 단계. 원인은 872001의 폰트 로더가
원본 크기 기준 **고정 버퍼**를 할당하는 것으로 추정(872000은 이 제약이 없어 문제
없었음).

**해법**: 실제 사용되는 일본어 코드포인트(MSG_PK+base MSG+exefs/main 문자열)와
사용되지 않는 전각 글자 슬롯을 계산해, **미사용 전각 글자의 비트맵을 한글로
제자리 치환**(charmap 재매핑만, 압축 해제 크기 완전히 그대로 유지). 이러면 872001의
버퍼 가정을 깨지 않아 정상 부팅. 게임이 실제 쓰는 문자 수(<7,000자) 대비 폰트가
보유한 총 글리프(7,115개)에 여유가 있어 가능했음(정책상 필요한 한글 1,260자 < 여유
3,000여 슬롯).
- 도구: `tools/g1n_inplace_korean.py`

### 자간·대사 줄바꿈 (§5의 자간 최적화, PUK에서 별도로 다시 필요했던 이유)
제자리 삽입된 한글 글리프는 대체 대상이었던 미사용 전각 글자의 메트릭(byte4=48,
안 조여진 값)을 그대로 물려받습니다. §5에서 이미 자간을 조인 일반 게임 폰트와
달리 PUK 폰트는 처음부터 다시 조여야 함 — 안 하면 대사가 훨씬 넓게 렌더링되어
3줄 박스를 넘겨 잘림. `tools/puk_font_tighten.py`가 §5와 동일한 알고리즘(잉크
폭 기반, byte4만 변경)을 적용하고, `tools/puk_msgev_fix.py`가 §4 말미의 대사
줄바꿈 최적화를 `msgev.bin`(PUK 대사)에 동일하게 적용합니다.
**주의**: byte4만 바꿔도 압축 크기는 변할 수 있음(엔트로피 변화) — 압축 해제
크기 불변은 유지되지만, 물리 슬롯(다음 엔트리 오프셋까지 남은 공간) 기준으로
여유가 있는지 반드시 확인 후 써야 함(§7 위 문단에서 이미 슬롯에 여유 생김).

### exefs/main
872000과 별개의 실행파일. 시스템 메시지 6종은 `tools/patch_main.py`를 872001의

### res_lang_pk.bin의 PUK 전용 이미지 (엔트리 18·21)
`res_lang_pk.bin`(27개 외부 엔트리) 중 18·19·20·21·22번은 §1의 내부 LINKDATA가
**여러 개의 단일-텍스처 자식**을 담은 컨테이너(§6의 e5/e8과 달리 한 엔트리에
텍스처 하나가 아니라, LINK 하나에 자식 수십 개 — 각 자식이 KT+G1T 하나씩).
- **엔트리18**: 자식 43개, 전부 512×64 — PUK 메뉴/툴팁 텍스트 라벨(정책·시설·편집
  등). 텍스트 활성 영역은 캔버스 좌상단 일부(예: 90×45px)뿐이고 나머지는 완전
  투명 — 캔버스 크기에 얽매이지 않고 활성 알파 bbox만 다뤄야 함.
- **엔트리21**: 자식 33개, 전부 1024×256 — 특성/가보 이름 배지(금색 글로우 텍스트).
  이름 하나당 렌더 상태 변형 3장(선명·선명·모션블러) 세트로 존재.
- 재조립: 자식 목록을 완전히 새로 구성(KT 재래핑 후 offset/size 재계산, 16바이트
  정렬) → 외부 엔트리 하나를 통째로 교체. **폰트 엔트리(16·17)와 달리 이들은
  일반 이미지 컨테이너라 크기 변경·오프셋 이동에 제약이 없음** (오프셋 고정이
  필요했던 건 §5에서 설명한 폰트 로더의 고정 버퍼 문제이지, 이미지 전반의 문제가
  아님 — 단, 실기에서 최종 검증은 필요).
- **⚠️ 투명 배경 텍스트에 `koloc.erase_place`를 그대로 쓰면 잔상이 남음**: 그
  함수는 RGB만 인페인팅하고 알파는 그대로 두는데, 이는 불투명한 배경색이 있는
  버튼(§6 e5/e8)에는 맞지만 투명 배경에는 인페인팅이 지울 게 없어 얼룩진 색이
  남고 알파도 그대로 불투명이라 잔상으로 보임. 알파 채널 자체를 0으로 지우는
  전용 함수가 필요함.
- 도구: `tools/loc_e18_labels.py`, `tools/loc_e21_badges.py`,
